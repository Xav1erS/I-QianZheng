### 1. 项目概述

#### 1.1 基本信息

| 属性 | 值 |
|------|-----|
| **产品名称** | 问签（wèn qiān） |
| **英文标识** | I-QianZheng |
| **副标题** | 你的 AI 签证顾问 |
| **域名** | iqianzheng.com |
| **部署平台** | Vercel |
| **当前阶段** | MVP 已完成，9 个页面，构建通过 |

#### 1.2 产品核心价值

**一句话定义：**
> 通过 AI 技术替代昂贵、不透明的传统签证顾问，让用户 3 分钟内获得个性化签证路径报告。

**核心价值三角：**

```
       AI 决策能力
          △
         / \
        /   \
       /     \
      /       \
决策咨询 ──── 表单自动化
（路径推荐）   （材料清单）
```

**解决的核心痛点：**
- 传统签证顾问收费高（数千至数万元）、不透明
- 签证政策信息分散、难以自查
- 个人情况差异大，通用攻略参考价值有限
- 初步筛选阶段不需要律师介入，但缺少低成本工具

#### 1.3 目标用户

**主要用户群：**
- 计划移民/留学的中国大陆用户（25-45岁，有一定收入）
- 东南亚华人（菲律宾、马来西亚、印度尼西亚等）
- 已有移民想法但不确定路径的用户（探索阶段）

**用户需求场景：**
1. 「我想移民澳洲，不知道走哪条路，先问问看」
2. 「准备申请美签，想提前了解成功率和材料」
3. 「跟签证中介谈之前，想先自己做功课」

**用户决策路径：**
```
问题意识 → 搜索信息 → 问签评估（3分钟）→ 获得报告 → 深度咨询/自行准备
```

#### 1.4 当前实现状态（MVP）

已完成的页面（`npm run build` 9 个页面通过）：

| 页面路由 | 功能 | 渲染方式 |
|---------|------|---------|
| `/` | 首页 | SSG |
| `/wizard` | 3步评估向导 | CSR |
| `/result/[id]` | AI 报告结果页 | CSR + 流式 |
| `/history` | 历史报告列表 | SSR |
| `/auth/login` | 登录/注册 | CSR |
| `/auth/callback` | Auth 回调 | Route Handler |
| `/api/analyze` | AI 分析 API | API Route |

### 2. 产品定位与竞品分析

#### 2.1 外部评估结论（ChatGPT 产品经理视角）

> 评估时间：2026年初，基于线上版本 iqianzheng.com

| 评估维度 | 评分 | 核心问题 |
|---------|------|---------|
| 产品方向 | **9/10** | 赛道成立，方向正确 |
| 网站表达 | **4/10** | 像传统中介，不像 AI 产品 |
| 商业潜力 | **8/10** | 多条变现路径，SaaS 化空间大 |

**核心矛盾：**
> 产品叫「AI 签证顾问」，但页面看起来像「签证中介」。

用户进入首页后的认知：
- **实际感受：**「这是签证代办网站」
- **期望感受：**「这是 AI 签证工具」

#### 2.2 竞品参考

| 产品 | 定位 | 特点 | 差距 |
|------|------|------|------|
| Jenova（国外） | AI 移民助手 | 资格评估 + 签证策略 + 材料准备 | 无中文，不针对中国用户 |
| 传统签证代办网站 | 人工代办 | 全流程服务，高收费 | 无 AI，不透明 |
| 移民局官网 | 政策信息 | 权威但复杂 | 无个性化，无路径推荐 |
| **问签（目标）** | **AI 路径顾问** | **中文，低价，个性化，3分钟出报告** | - |

#### 2.3 问签的差异化定位

**不是什么：**
- ❌ 不是签证代办（不帮用户递交申请）
- ❌ 不是法律顾问（免责声明明确）
- ❌ 不是移民律师平台

**是什么：**
- ✅ AI 驱动的签证路径评估工具
- ✅ 决策前的低成本信息获取渠道
- ✅ 「先了解选项，再咨询律师」的第一步

#### 2.4 改进优先级矩阵

```
高影响力
    ↑
    │  首页 AI 感重构 ★★★  │  面签训练模块
    │  报告示例展示  ★★★   │  支付系统集成
    │                        │
────┼────────────────────────┼────→ 高实施成本
    │                        │
    │  积分 Toast 优化        │  多语言支持
    │  定价区块首页            │  企业版
    ↓
低影响力
```

### 3. 商业化路径规划

#### 3.1 当前变现模式

- **注册即赠 3 次免费评估**（无需信用卡）
- **后续付费积分**（价格待定）

设计原则：先让用户体验价值，再付费转化。

#### 3.2 四条变现路径（ChatGPT 建议）

| 路径 | 产品形态 | 参考定价 | 优先级 | 状态 |
|------|---------|---------|--------|------|
| **AI 路径报告** | 3步向导 → Markdown 报告 | 9.9–49 元/次 | P0 | ✅ 已实现（积分扣除） |
| **AI 材料生成** | 行程计划 / 解释信 / 动机信生成 | 19–99 元/份 | P1 | 🔲 待开发 |
| **AI 面签训练** | 模拟面试问答（对话式） | 39 元/次 | P2 | 🔲 待开发 |
| **人工顾问对接** | 平台预约持牌律师/顾问 | 999–5000 元 | P3 | 🔲 规划中 |

#### 3.3 短期目标（3个月内）

1. 完成 P1 首页重构，提升 AI 产品感 → 提高注册转化率
2. 接入支付（微信/支付宝），实现积分付费购买
3. 推出「AI 材料生成」功能（解释信/行程计划）
4. 建立邮件列表（注册用户通知）

#### 3.4 中期目标（6-12个月）

1. 月活用户 5000+
2. 付费转化率 10%+
3. 推出面签训练模块
4. 探索与移民律师的佣金分成合作模式

### 5. 非功能需求

#### 5.1 性能要求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| AI 流式首字节延迟 | < 3 秒 | 从提交到看到第一个字 |
| 页面 TTI（首屏） | < 2 秒 | 首页/登录页 |
| 历史页 SSR 响应 | < 500ms | Vercel Edge Network |
| AI 报告完整生成 | 30–60 秒 | Vercel maxDuration=60 |

#### 5.2 安全要求

| 要求 | 实现方式 |
|------|---------|
| 用户数据隔离 | Supabase RLS（全表覆盖） |
| Service Role Key 保密 | 仅服务端使用，永不暴露前端 |
| 禁止文件上传 | 设计层面不提供上传功能 |
| API 认证 | JWT Bearer Token（Supabase Auth） |
| XSS 防护 | React 默认转义 + ReactMarkdown 受控渲染 |
| 积分篡改防护 | 积分扣除在服务端执行，前端无写权限 |

#### 5.3 可用性要求

- **移动端优先**：所有页面在 iPhone SE（375px）宽度正常可用
- **浏览器兼容**：iOS Safari 15+、Chrome 100+、Edge 100+
- **无障碍**：关键交互元素有 `aria-label`，键盘可操作
- **错误处理**：所有 API 错误有用户友好的 Toast 提示

#### 5.4 合规要求

- **双重免责声明**：
  - 层级 1：System Prompt 中明确告知 AI 输出的局限性
  - 层级 2：前端每个报告页顶部固定黄色警告横幅
- **不存储证件**：产品设计层面不允许文件上传
- **法律建议声明**：所有报告明确标注「不构成法律建议」
- **隐私说明**：首页注明「无需上传任何证件，隐私优先」

### 9. 核心数据流程

#### 9.1 完整用户旅程（Happy Path）

```
1. 用户访问首页 /
        ↓
2. 点击「开始 AI 评估」→ 跳转 /wizard
        ↓（未登录 → 中间件拦截 → 跳转 /auth/login?redirect=/wizard）
3. 注册账号（邮箱 + 密码）→ 收到验证邮件 → 点击确认
        ↓
4. 自动赠送 3 积分（数据库触发器 handle_new_user）
        ↓
5. 返回 /wizard，填写 Step 1（用户画像）
        ↓
6. 填写 Step 2（职业背景）
        ↓
7. 填写 Step 3（移民意向）→ 点击「生成评估报告」
        ↓
8. POST /api/analyze
   - 服务端验证 JWT → 查询积分 → 创建 consultation 记录（ai_response 为 NULL）
   - 返回 { consultationId }
        ↓
9. 前端跳转 /result/[consultationId]
        ↓
10. 结果页发起 AI 流式请求
    - 显示蓝色状态条「AI 正在生成报告...」
    - 实时累积流式内容（whitespace-pre-wrap + 游标动画）
        ↓
11. 流式完成
    - 后端保存完整 ai_response 到数据库
    - 后端扣除 1 积分（credits - 1）
    - 前端切换为 ReactMarkdown 完整渲染
    - Toast：「报告生成完毕！」
        ↓
12. 用户查看报告（可复制 / 导出 PDF）
        ↓
13. 访问 /history 查看历史报告列表
```

#### 9.2 积分不足流程

```
Step 3 → 点击提交 → POST /api/analyze
        ↓
服务端：credits = 0 → 返回 402 Forbidden
        ↓
前端：Toast 错误「积分不足，请联系我们获取更多积分」
        ↓
引导用户（未来：购买积分）
```

#### 9.3 流式传输技术细节

```
前端（result/page.tsx）
    ↓ fetch('/api/stream', { method: 'POST' })
    ↓ reader = response.body.getReader()
    ↓ while(true) { const { done, value } = await reader.read() }
    ↓ 累积 content 字符串，触发 setState 重渲染
    ↓ done = true → setIsStreaming(false) → ReactMarkdown 渲染

后端（api/analyze/route.ts）
    ↓ anthropic.messages.stream(...)
    ↓ new ReadableStream({ start(controller) { ... } })
    ↓ 每收到 chunk → controller.enqueue(chunk)
    ↓ stream.finalMessage() → 保存到 DB → 扣积分
    ↓ controller.close()
```

#### 10.4 RLS 策略

```sql
-- users 表
CREATE POLICY "users_select_own" ON users
  FOR SELECT USING (auth.uid() = id);

CREATE POLICY "users_update_own" ON users
  FOR UPDATE USING (auth.uid() = id);

-- consultations 表
CREATE POLICY "consultations_select_own" ON consultations
  FOR SELECT USING (auth.uid() = user_id);

CREATE POLICY "consultations_insert_own" ON consultations
  FOR INSERT WITH CHECK (auth.uid() = user_id);
```

> ⚠️ 积分（credits）的扣减通过 `SUPABASE_SERVICE_ROLE_KEY` 在服务端执行，绕过 RLS，防止用户通过 API 直接操控积分。

#### 10.5 触发器

```sql
-- 新用户注册后自动创建 users 记录（赠 3 积分）
CREATE OR REPLACE FUNCTION handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.users (id, email, credits)
  VALUES (NEW.id, NEW.email, 3);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION handle_new_user();
```

#### 10.6 索引优化

```sql
CREATE INDEX idx_consultations_user_id ON consultations(user_id);
CREATE INDEX idx_consultations_created_at ON consultations(created_at DESC);
```

### 11. API 接口定义

#### 11.1 POST /api/analyze

**用途：** 创建咨询记录占位，返回 consultationId，前端凭此 ID 发起 AI 流式请求。

| 属性 | 值 |
|------|-----|
| Method | POST |
| Path | `/api/analyze` |
| 认证 | Bearer JWT（Supabase Auth） |
| Content-Type | `application/json` |
| maxDuration | 60秒（Vercel 配置） |

**请求 Body：**
```typescript
type AnalyzeRequest = WizardFormData; // 见 types/index.ts

interface WizardFormData {
  nationality: string;
  targetCountries: string[];
  age: number;
  education: string;
  career: string;
  income: string;
  hasSpouse: boolean;
  hasChildren: boolean;
  purpose: string;
  budget: string;
  willInvest: string;
}
```

**成功响应（200）：**
```json
{
  "consultationId": "uuid-string"
}
```

**错误响应：**

| HTTP 状态码 | 原因 | 前端处理 |
|-----------|------|---------|
| 401 | 未登录/Token 过期 | Toast 提示 → 跳转登录页 |
| 402 | 积分不足 | Toast 提示「积分不足」 |
| 422 | 表单数据不完整 | Toast 提示「请完整填写信息」 |
| 500 | 服务器错误/AI API 异常 | Toast 提示「服务异常，请稍后重试」 |

**服务端执行流程：**
```
1. createServerClient(cookies) → 获取当前用户
2. 若未认证 → return 401
3. supabase.from('users').select('credits').eq('id', user.id)
4. 若 credits <= 0 → return 402
5. 验证 formData 字段完整性
6. createAdminClient().from('consultations').insert({...})
7. return { consultationId }
```

> **设计决策：** `/api/analyze` 仅创建记录并返回 ID，不触发 AI。AI 流式由结果页单独发起，便于前端控制展示状态。

#### 11.2 AI 流式端点（内部）

流式调用在 `result/[id]/page.tsx` 的客户端 useEffect 中直接发起，通过相同的 `/api/analyze` 路由或独立流式路由处理。

**流式响应格式：**
```
Content-Type: text/event-stream
Transfer-Encoding: chunked

data: {"chunk": "推荐"}
data: {"chunk": "签证"}
data: {"chunk": "路径"}
...
data: [DONE]
```

| 直接访问不存在的 /result/[id] | 404 页面或跳转历史页 |
| Safari iOS 兼容性 | 流式读取、PDF 打印正常 |

#### 16.3 关键配置注意事项

| 注意点 | 说明 |
|--------|------|
| `next.config.mjs` | Next.js 配置文件不支持 `.ts` 扩展名，必须用 `.mjs` |
